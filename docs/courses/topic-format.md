# トピック教本 Markdown フォーマット

JukuBox のコースを作る際に、各トピックの教本 Markdown が満たすべきフォーマットを定める。
本ドキュメントは**コース作成者向け**であり、講師ロール（jukubox スキル）が学習者を進行させる前提条件となる。

## セクション構成

トピック Markdown は以下のセクションで構成する。

| セクション         | 見出しレベル | 必須 / 任意 | 役割                                                               |
| :----------------- | :----------- | :---------- | :----------------------------------------------------------------- |
| トピックタイトル   | `#`          | 必須        | トピックの主題を端的に表す                                         |
| `## 目標`          | `##`         | 必須        | このトピックを終えたとき学習者が到達している状態                   |
| `## 知識`          | `##`         | **必須**    | ステップで実践する内容を理解するための前提知識                       |
| `## ステップ`        | `##`         | 必須        | 学習者が手を動かして実行する手順。`###` のステップに分割する   |
| `## 完了判定`      | `##`         | 必須        | トピック完了の判定基準（客観的・主観的どちらでも可）               |
| `## 補足`          | `##`         | 任意        | つまずきやすいポイント、エラー時の対処、発展的な情報               |
| `## 理解度チェック` | `##`         | 任意        | 知識の定着を確認する設問。概念暗記が不要なトピックでは省略してよい |

## 必須セクションの理由

### 知識セクションが必須である理由

講師ロールは「**まず知識セクションの内容を学習者のレベルに合わせて解説 → ステップに移る**」という流れで進行する。
知識セクションが存在しない場合、講師は前提知識のないままステップへ入ることになり、学習者が「なぜこの操作をするのか」を理解できないまま手順だけをなぞる状態になる。

このため、たとえ環境構築のように手順が中心のトピックであっても、最低限の前提知識（使うツールの位置づけ、操作の目的など）を知識セクションに記述する。

### ステップセクションの分割ルール

- ステップは `###` 見出しで区切る
- 各ステップは「期待される成果物」（生成・編集されるファイル、設定値、ディレクトリ構造など）が明確になるように書く
  - これは講師ロールがレジューム時に進捗を判定するための情報源になる
- 学習者の応答や作業が必要なステップと、解説のみで一方通行に進めてよいステップを意識して書き分ける

## 推奨される書き方

- 知識セクションは「ステップで実際に試す範囲に必要な分量」に絞る。長すぎると解説中に学習者が忘れるため、深掘りは補足セクションへ回す
- 各ステップの見出しには動詞を含めて何をするかを明確化する（例: `### 1. LikeButton を作る`）
- 完了判定は客観的に確認できるもの（ファイル存在、コマンド出力）と主観的なもの（自分の言葉で説明できる等）を分けて書く
